# /execute

コンテキストは必ず失われる。PLAN.md は新しいセッションの自分が読んで即再開できるための外部記憶であり、コミットは進捗が消えないための保険である。

## Advisor（設計判断）

以下の場面では **Advisor を opus モデルで呼び出し**、返ってきた設計をそのまま使う。
自分（Executor）で設計判断をしない。

**呼び出す場面：**
- PLAN.md が存在しない（初回プラン生成）
- ユーザーから要件変更・フェーズ追加の指示があった
- アーキテクチャの選択肢が複数あり自分では決めかねる

**呼び出し方：**

```
Agent(
  model: "opus",
  prompt: "[advisor スキルの指示内容] + [現在の状況・要件・既存 PLAN.md の内容]"
)
```

Advisor はコードを書かず判断のみ返す。Executor はその出力を PLAN.md に書き込む。

---

## 起動時の動作

### PLAN.md が存在しない場合 → ヒアリング → Advisor

以下を順番に確認してください（一度に全部聞かず、会話しながら進める）：

1. 何を作りたいか（概要）
2. 動作環境（OS・言語・主要ライブラリ）
3. 主な要件・操作仕様
4. 制約・決定事項（すでに決まっていることがあれば）

ヒアリングが終わったら **Advisor を呼び出し**、返ってきたプランをもとに以下のフォーマットで `PLAN.md` を生成してください：

```markdown
# {プロジェクト名} - Implementation Plan

## プロジェクト概要
{一文で}

## 環境
- {OS / ハードウェア}
- {言語・フレームワーク}
- {主要ライブラリ}

## 操作仕様
- {要件を箇条書き}

## 受け入れ条件
- {このプロジェクトで「動いた」とみなす基準}

## 完了条件
- {各タスク完了時に実行するものの箇条書き。選択的に実行する}

---

## 🔥 Hotfix（最優先）

<!-- 動作確認中の不具合・緊急対応はここに積む。未完了がある限り最優先で対応する -->

---

## Phase 1: {フェーズ名}

- [ ] 1-1. {タスク}
- [ ] 1-2. {タスク}

## Phase 2: {フェーズ名} [REVIEW]

<!-- [REVIEW] をつけた Phase は全タスク完了後に自動レビューを実行する -->

- [ ] 2-1. {タスク}

---

## メモ・決定事項
- {設計上の判断とその理由}

## 完了済みフェーズ
<!-- Phase {N}: {フェーズ名} `{開始ハッシュ}..{終了ハッシュ}` -->
```

生成後、「このプランで進めますか？」と確認を取ってください。

---

### PLAN.md が存在する場合 → 再開フェーズ

1. `PLAN.md` を読み込む
2. PLAN.md の構造をこのスキルのテンプレートと照合し、不足セクションは補完・テンプレートにないセクションは削除してコミットする
3. **Hotfix セクションに未完了タスクがあればそちらを最優先**で開始する
4. Hotfix がなければ通常 Phase の最初の未完了タスク（`- [ ]`）を特定する
5. 「{タスク番号}. {タスク名} から再開します」と報告する
6. 実行を開始する

---

## 実行ルール

1 タスクを完了するたびに、必ず以下を行います：

1. **フォーマットを実行する**（CLAUDE.md のフォーマット規約に従う）
2. `PLAN.md` の該当チェックボックスを `- [x]` に更新する
3. git commit する（メッセージは完了したタスク内容を端的に）
4. `git remote` でリモートが存在する場合のみ `git push` する（`debug/` ブランチはスキップ）。トラッキングブランチ未設定なら `git push -u origin <branch>` で設定してから push する
5. レビュー判定を行う（後述）
6. 問題なければ次のタスクへ自動進行する

**コミットメッセージの形式：**
```
{タスク番号}: {タスクの内容}

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

**フェーズ完了時：**
- PLAN.md のフェーズ詳細を削除し、完了済みフェーズセクションに以下の形式で記録する
- `- Phase {N}: {フェーズ名} \`{開始ハッシュ}..{終了ハッシュ}\``
- 詳細は `git log {開始ハッシュ}..{終了ハッシュ}` で追える

## タスク・フェーズの分割

**タスク:** 1 コミットで意味が完結する単位。「ファイル1つ」「関数1つ」「設定1項目」など、何をしたか一文で言える粒度にする。

**フェーズ:** 技術的にまとまりのある作業のまとまり。「このフェーズが終わると何ができるか」を軸に切る。フェーズに `[REVIEW]` をつけることでレビューポイントにできるが、すべてのフェーズがレビュー対象である必要はない。

---

## レビュー判定ルール

以下のいずれかに該当する場合はレビューを実行する：

1. **`[REVIEW]` マーク** — Phase ヘッダーに `[REVIEW]` がついており、その Phase の全タスクが完了したとき
2. **Claude 自身が確認を求めたいとき** — 実装の方向性に迷いがあり、続ける前にユーザーの判断が必要なとき

### 自動レビュー（`[REVIEW]` フェーズ完了時）

`review` スキルを呼び出して Security・DevOps の2観点から自動チェックを実行する。

- 🔴 要修正（high severity）→ 修正タスクを PLAN.md の Hotfix セクションに追記してコミットし、修正を優先する
- 🟡 要確認（medium/low）→ 指摘内容をユーザーに提示し、判断を仰ぐ
- 🟢 問題なし → ユーザーにフェーズ完了を報告し、次フェーズへ自動進行する

**ユーザーへの報告フォーマット（🟢 または 🟡 のとき）：**
```
👀 レビュー完了 — {Phase 名}

{review スキルの結果}

{🟢 問題なし → 「次の Phase へ進みます」 | 🟡 → 「上記を確認してください。続けますか？」}
```

### 手動レビュー（Claude 自身が確認を求めるとき）

```
❓ 確認が必要です — {理由}

{確認内容}

問題なければ「続けて」、修正があればお知らせください。
```

## /investigate への移行

以下のいずれかに該当するとき、`/investigate` スキルへ移行する：

- コードや現在見えているログだけでは原因を特定できない
- ユーザーと Claude の認識に齟齬があり、コードを読んでも解消されない
- 「何が起きているか」を把握するために実際の実行ログが必要と判断した

移行時はユーザーに宣言する：
```
🔍 ログの確認が必要と判断しました。/investigate で調査します。
```

`/investigate` 収束後、Hotfix セクションの修正プランを元に実行を再開する。

---

## 注意事項

- 複数タスクをまとめて実行しない
- タスクをまたいだ大きなリファクタリングは提案するがすぐ実行しない
- 要件が不明確なまま実装を始めない。わからないことは勇気を持って聞く
- 全タスク完了時は「🎉 全タスク完了です」と報告する

**実装はすべて PLAN.md を経由する：**
- 動作確認中の不具合・追加実装が発生したら、コードに手をつける前に必ず `PLAN.md` に追記する
- 緊急・高優先度のものは Hotfix セクションへ、通常のものは該当 Phase へ
- PLAN.md への追記→コミット→実装、の順を崩さない
